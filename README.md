# 1- Retomada da aula anterior 
Ciclo de vida do desenvolvimento de software:
- Primeira fase- levantamento de requisitos.
- Segunda fase- análise dos requisitos.
- Terceira fase- Modelagem.
- Quarta fase- Codificação.
- Quinta fase- Testes.
- Sexta fase- Implementação.
- Sétima fase- Manutenção.

Modelos: 
- Modelo cascata: simples e fácil de entender e executar, funcional para sistemas que não mudam requisitos pois é demorado.
- Modelo espiral: modelo interativo e progressivo, feito em fases (ciclos, podendo voltar para os ciclos anteriores), muita interatividade aumenta as fases e tempo.
- Modelo incremental: entregas em partes menores, tendo uma interação melhor com as entregas.

Manifesto ágil 
- Metodologia: jeito de organizar e conduzir um projeto do começo ao fim.
- Framework: uma estrutura pronta que ajuda a organizar e facilitar o desenvolvimento.
- Método: passo a passo específico para fazer uma tarefa.




# 2. Técnicas de levantamento de requisitos

| Técnica | Descrição | Quando Usar | Vantagens | Limitações |
|---------|-----------|-------------|-----------|------------|
| Entrevista | Uma conversa direcionada com um propósito específico, que utiliza um formato de perguntas e respostas | Usada quando é necessário coletar informações detalhadas e necessidades especificas. | Maior profundidade, mais interatitvo, tendo uma interação mais direta. |  É necessário muito tempo para realizar as entrevistas além de contar com a habilidade do entrevistador para conseguir de forma concreta oque o cliente deseja. |
| Questionário | Conjunto de perguntas enviadas para várias pessoas para coletar informações e entender o que elas precisam do sistema. | Grande número de pessoas envolvidas no projeto do sistema e é necessário saber a quantidade de pessoas que aprovam determinado dado/situação/característica do sistema. | Útil para coletar informações de um grupo muito grande de pessoas, sendo a forma mais rápida. | Como é um grande número de pessoas respondendo pergunta de forma simples, acaba não tendo uma profundidade tão grande |
| Observação  | Observar o comportamento e o ambiente da pessoa é eficaz em levantar informações que passam despercebidas quando outras técnicas são usadas. | Usada quando os usuários tem dificuldade em verbalizar em uma entrevista por exemplo, e também para detectar requisitos mais implicitos. | Capta o comportamento real e natural do usuário em seu ambiente. | Consome maior tempo e pode levar a erros do analista, levar a uma interpretação equivocada. |
| Análise de documentos  | Análise de documentos já feitos para entender como o sistema tem que funcionar | Muito utilizado quando a empresa/cliente já possui registros, documento, etc. já definidos, ou um sistema antigo já feito | Provável de ter informações mais concretas e detalhadas, e reduz a quantidade de entrevistas e encontro com o cliente | Podem estar desatualizados e incompletas, principalmente se forem documentos muito mais antigo |
| Brainstorming  | Nesse tipo de reunião, representantes interessados realizam uma discussão informal para gerar tantas ideias. | Recomendado quando o projeto está em fase muito inicial e é preciso explorar idéias para ter um norte de onde seguir | Estimula muito a criatividade, e por ser uma conversa mais informal acaba dando mais espaço para os participantes interagirem mais | Sessão mal conduzida pode gerar “atropelo de idéias”, e alguns participantes falando bem mais que outros |
| JAD — Joint Application Development  | Participa diferentes interessados (analistas, clientes, dev, etc) na reunião por meio de discussões, tanto de problemas a serem resolvidos quanto das soluções disponíveis | Utilizada principalmente quando um projeto é mais complexo, envolvendo várias áreas | Maior clareza nas decisões, melhor comunicação entre os envolvidos e maior comprometimento dos participantes com o projeto | Por ter que reunir todos os participantes pode acabar sendo inviável, teno que ter um ótimo planejamento para funcionar |
| Prototipagem  | Cria um protótipo do sistema que é desenvolvida no início do processo de desenvolvimento para que os usuários possam ter uma visão incial de como vai funcionar. | Recomendada quando o usuário/cliente não consegue deixar muito claro em entrevista os requisitos para o sistema. | Melhor compreensão dos requisitos, reduz problemas e participação forte do usuário | Pode gerar aumento de custo e tempo se houver muitas mudanças.|
| Estudo Etnográfico  | É baseada na observação direta do ambiente de trabalho dos usuários, analisando como eles realmente fazem suas atividades no dia a dia | Indicada quando é importante entender muito bem o ambiente e o comportamento dos usuários, principalmente em sistemas que envolvem muitas atividades práticas | Identificação de problemas ocultos, compreender o contexto social e a organização | Preicsa de muito tempo, e tem que contar com a habilidade de quem observa para diminuir erros. |
